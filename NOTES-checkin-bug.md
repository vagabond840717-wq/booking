# 체크인 날짜 밀림 버그 — 조사 메모

> 새 세션에서 이어서 작업하기 위한 인수인계 노트. (작성일 2026-07-02)

## 증상
- **603호 부킹닷컴**: 실제 예약 `6/21 ~ 7/3 (12박, CHOLONG KIM, 예약번호 6297093725)`
  인데 앱 달력엔 체크인이 **7/2**로 표시됨.
- **402호 트립닷컴**: 실제 예약 `6/27 ~ 7/4 (7박, DING/QIAN, 예약번호 1128148388218993)`
  인데 앱 달력엔 체크인이 **7/1**로 표시됨.
- 두 방 다 **퇴실일은 정확**, **체크인만 뒤로 밀림**.

## 원인 (진단 완료)
1. **앱(index.html) 잘못이 아님.** `cellTypeFor()`(index.html 1016~1031줄)는
   체크인 셀('인')을 booking의 `cinY/cinM/cinD`가 그 날짜일 때만 그림. 즉 앱이 받은
   데이터의 체크인 자체가 7/2·7/1로 들어와 있어서 그렇게 그려지는 것.
2. 예약을 **수정/연장**하면 OTA(부킹·트립) iCal이 한 예약을 **여러 토막(VEVENT)**으로
   쪼개서 내보냄. (402호는 화면에 "6/23 숙소가 변경 요청 수락 → 기존 예약 업데이트"
   기록이 실제로 찍혀 있었음.)
3. 서버 worker의 `parseIcal`에 **"체크아웃이 오늘 이전이면 제외"** 필터가 있음
   (`worker_backup_0524.js` 318~321줄). 그래서 아직 투숙 중인 예약의 **앞 토막
   (late 6월 부분)**이 "이미 퇴실 지난 예약"으로 오인되어 버려지고, 남은 뒷 토막의
   시작일이 새 체크인처럼 보임.
   - 603: 6/21 앞토막 버려짐 → 남은 토막 시작 7/2가 체크인으로 표시
   - 402: 6/27 앞토막 버려짐 → 남은 토막 시작 7/1이 체크인으로 표시
   - 퇴실일(7/3·7/4)이 정확한 것도 이 설명과 일치.

### 확인 근거
- worker `parseIcal`는 DTSTART를 시간대 보정 없이 그대로 파싱(`d.slice`)하므로
  단일 VEVENT였다면 6/21이 그대로 유지됨 → 앱도 6/21로 그렸을 것.
- 앱이 7/2를 그리려면 데이터에 cin=7/2 booking이 있어야만 함 → 피드가 다토막이고
  worker가 앞토막을 버렸다는 뜻. (내부 정합성 확인됨)
- `syncAllRooms`(worker 210~265줄)는 `synced_bookings`를 매번 덮어씀(누적 아님).
  프런트의 `mergeArchiveIntoRooms`는 과거(cout<now) 예약만 추가 → 미래/오늘 체크인
  셀을 만들 수 없음. 따라서 프런트가 만든 게 아님.

## 고칠 곳
- **앱(index.html)이 아니라 서버 worker** (`ical-proxy.vagabond1984.workers.dev`).
  앱이 데이터를 받는 시점엔 이미 앞 토막이 지워져 있어 **앱만으로는 6/21을 되살릴 수 없음.**
- 저장소엔 `worker_backup_0524.js`(구버전 백업)만 있음. 실제 배포는 Cloudflare(wrangler)로
  별도. 백업 파일이 배포본과 같다는 보장 없음 — 배포본 확인 필요.

## 수정 방향
- `parseIcal`에서 **같은 예약의 토막을 이어붙인 뒤** 과거 필터를 적용.
- ⚠️ **주의**: 날짜만 보고(앞 토막 cout == 뒷 토막 cin) 무작정 이어붙이면 *진짜로
  손님이 연달아 바뀐 경우*(A 퇴실 7/1 → B 입실 7/1)까지 한 명으로 합쳐 회전을 놓침.
  → **예약 고유번호(UID) 또는 SUMMARY로 같은 예약인지 확인**해서 이어붙여야 안전.
  → 그러려면 실제 iCal 피드 원본(부킹 `bkUrl` / 트립 `trUrl`)의 VEVENT 구조를
    한 번 봐야 함.

## 진행하려면 필요한 것 / 걸림돌
1. **서버 접속**: 이전 세션에선 egress 정책상 `ical-proxy...workers.dev` 접속이
   403으로 차단됨. 새 세션이 이 도메인에 접속되는지부터 확인할 것. 막혀 있으면
   iCal 링크나 원본 몇 줄을 붙여넣어 주면 오프라인으로 수정안 작성 가능.
2. **배포 방법**: worker는 이 저장소가 아니라 Cloudflare 배포. wrangler로 사장님이
   올리거나 배포 경로 공유 필요.
